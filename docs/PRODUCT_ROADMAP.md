# M記 产品演进方案：从「写作器」到「可问答的本地知识库」

文档状态：草案  
关联文档：[PRD](./PRD.md)  
基线版本：v0.9.12 / 1.0 发布候选  
更新时间：2026-07-22

## 1. 洞察与命题

2026 年用户听到「AI 知识库」时，默认期待的是：

> 对着整个本地库提问，答案带来源，并能跳回原文。

M記 当前强项是 **本地 Markdown 写作与发布闭环**（编辑–预览一致、Wiki/附件、导出、simple-blog / 微信草稿、可选单篇 AI 辅助）。缺口不在「再多几个润色按钮」，而在 **把本地 Markdown 库变成可检索、可问答、可引用的知识层**。

因此后续演进的核心命题是：

**在不牺牲本地优先、文件透明、写作优先的前提下，补齐「库级检索增强 AI」与「知识关系发现」，把 M記 从「带 AI 的 Markdown 写作器」升级为「可问答的本地知识库写作工具」。**

## 2. 演进原则（不可破）

与 [PRD](./PRD.md) §2 / §5.3 保持一致：

| 原则 | 含义 |
| ---- | ---- |
| 本地优先 | 索引、缓存可本地生成；笔记与附件仍是用户文件夹里的普通文件 |
| 文件透明 | 不引入封闭专有库作为内容本体；索引损坏可重建，不影响 `.md` |
| 写作优先 | AI / 检索失败不得阻塞编辑、保存、预览 |
| AI 可选 | 无 API Key、无本地模型时，基础写作与发布完整可用 |
| 用户确认 | AI 改写、写回文件、上传内容前必须显式确认 |
| 刻意不做 | 云同步服务、多人协作、插件市场、富文本优先、企业连接器 / ACL |

判断标准：若某能力要求账号体系、专有云库或协作权限模型，默认拒绝，除非单独改定位。

## 3. 目标体验（North Star）

用户完成一次「知识库问答」闭环：

1. 打开本地知识库并正常写作（现有能力）。
2. 对知识库提问：「我们上次关于 xxx 的结论是什么？」
3. 得到带引用的答案，点击引用跳到对应笔记段落。
4. 在写作时看到「相关笔记」推荐，或通过反向链接发现被引用关系。
5. 可选：用本地模型完成上述流程，内容不离开本机。

一句话产品演进定位：

> **本地优先的 Markdown 知识库写作工具——写得顺，找得到，问得清。**

## 4. 分阶段演进

阶段按能力深度划分，不绑定日历工期。每一阶段应可独立发布、可独立验收。

### Phase 0 — 稳固 1.0 写作底座（进行中）

**目标：** 完成 PRD 1.0 门槛，保证写作/导出/发布可靠。

**范围：** 安装与更新、保存可靠性、预览一致性、发布成功率、中小型库性能。

**为何必须先做：** 知识层能力建立在稳定的文件树、搜索与 Markdown 解析之上；底座不稳，索引与问答会放大故障面。

**退出标准：** 达到 PRD §6 核心指标与发布候选检查清单。

---

### Phase 1 — 知识关系层（Backlinks First）

**目标：** 让「知识库」从文件夹浏览升级为可发现的笔记网络。

**能力：**

| 能力 | 说明 | 优先级 |
| ---- | ---- | ------ |
| 反向链接面板 | 展示「哪些笔记链接到当前笔记」；支持跳转 | P0 |
| 出链列表 | 当前笔记中的 `[[wiki]]` / 嵌入一览 | P0 |
| 未解析链接提示 | 死链可见，可一键创建目标笔记 | P1 |
| 轻量图谱（可选） | 当前笔记邻域图，不做全库力导向大图 | P2 |

**技术要点：**

- 基于现有 Wiki 解析，建立 vault 级链接索引（文件路径 ↔ 链接目标）。
- 索引增量更新：监听文件增删改；全量重建作为兜底。
- 索引落本地缓存目录（如知识库旁或应用数据目录），可删除重建。

**验收：**

- 打开笔记可在 1 秒量级看到反向链接（中小型库）。
- 重命名/移动文件后，索引最终一致；失败有重建入口。
- 无反向链接时 UI 不空转误导（明确「暂无」）。

**刻意不做：** 复杂图谱分析、Canvas、对象类型系统。

---

### Phase 2 — 本地检索增强（Semantic Find）

**目标：** 在关键词搜索之外，补上「意思相近也能找到」。

**能力：**

| 能力 | 说明 | 优先级 |
| ---- | ---- | ------ |
| 语义搜索入口 | 与现有关键词搜索并列或可切换 | P0 |
| 相关笔记推荐 | 写作/阅读时展示语义相近笔记 | P0 |
| Embedding 索引 | 本地向量索引；模型可配置（云或本地） | P0 |
| 混合排序 | 关键词命中 + 语义相似度融合 | P1 |

**技术要点：**

- 切片策略：按标题/段落/frontmatter 切块，保留路径与行号/锚点。
- 默认优先本地 embedding（若用户配置）；否则允许云端 embedding，但需设置页明确告知「会上传文本片段」。
- 与 Phase 1 链接索引解耦：检索层可独立开关。

**验收：**

- 用近义表述能找到相关笔记（准备固定评测集 20 问）。
- 未配置 embedding 时降级为关键词搜索，不报致命错误。
- 大库首次建索引有进度与可取消；不阻塞编辑。

**对标心智：** Obsidian Smart Connections 的「发现」能力，而非完整 Chat。

---

### Phase 3 — 库级问答（Ask the Vault）

**目标：** 兑现「AI 知识库」的核心承诺：问库、引用来源、跳回原文。

**能力：**

| 能力 | 说明 | 优先级 |
| ---- | ---- | ------ |
| Ask Vault 面板 | 自然语言提问 → 检索 → 生成答案 | P0 |
| 来源引用 | 每条关键答案附笔记路径 + 片段；可点击打开并定位 | P0 |
| 上下文范围 | 可选：全库 / 当前文件夹 / 当前笔记 / 手动点选笔记 | P0 |
| 对话历史 | 按知识库保存问答历史（本地 Markdown 或 JSON） | P1 |
| 基于答案继续写作 | 「插入到光标」「生成新笔记草稿」需确认 | P1 |
| 现有单篇 AI | 润色/摘要/标签/选区词条保留，并复用同一 Provider 配置 | P0 |

**交互约束：**

- 默认展示引用；无检索命中时明确说明，避免空口幻觉。
- 发送前可预览「将发送哪些片段」（隐私可见性）。
- 写回编辑器必须走确认流（沿用现有 AI Result Review）。

**技术要点：**

- RAG 管线：Query →（可选改写）→ Hybrid Retrieve → Rerank（可后置）→ Prompt with citations → LLM。
- Provider：延续 DeepSeek / Gemini / OpenAI 兼容；增加「仅用检索、不调用 LLM」的调试模式。
- 提示词：强制「只依据给定片段作答；不知则说不知；标注来源编号」。

**验收：**

- 固定 20 个「库内可验证问题」命中率与引用正确率可度量。
- 引用点击能打开正确文件并尽量定位到段落。
- 无 Key 时 Ask Vault 不可用，但不影响写作；提示引导配置。

**对标心智：** Obsidian Copilot Vault QA / NotebookLM 的「有来源问答」，但范围限定为本地 vault，不做多模态研究工作室。

---

### Phase 4 — 本地模型与隐私闭环

**目标：** 让「本地优先」在 AI 层也成立。

**能力：**

| 能力 | 说明 | 优先级 |
| ---- | ---- | ------ |
| Ollama / LM Studio / OpenAI 兼容本地端点 | 聊天与（可选）embedding | P0 |
| 隐私模式 | 禁止云端 Provider；仅本地索引 + 本地模型 | P0 |
| 发送内容清单 | 每次请求可查看将上传/本地推理的片段 | P1 |
| 索引不含敏感路径规则 | 可排除目录（如 `.trash`、私密文件夹） | P1 |

**验收：**

- 断网 + 本地模型下，语义搜索与 Ask Vault 仍可用（能力受模型质量约束，但链路通）。
- 隐私模式开启后，任何云端请求被拦截并提示。

---

### Phase 5 — 增强与扩展（按需，非承诺）

在 Phase 1–4 稳固后再评估，避免过早摊薄：

| 方向 | 说明 | 前提 |
| ---- | ---- | ---- |
| PDF/附件文本进入索引 | 预览已有 PDF，可抽文本进检索 | Phase 2/3 稳定 |
| 多知识库切换下的索引隔离 | 每库独立索引与历史 | 多库用户反馈明确 |
| 发布渠道扩展 | 更多博客/平台 | 写作用户增长 |
| 大型库性能 | 增量索引、后台队列、搜索延迟优化 | 用户库规模上来 |
| 邻域图谱增强 | 从 P2 轻量图升级 | Phase 1 使用率高 |
| 图床批量迁移 | PRD 已列 | 独立于 AI 主线 |

**明确后置 / 默认拒绝：**

- 企业连接器（Slack/Drive/Notion 同步索引）
- 团队权限与知识校验工作流（Guru 类）
- 对象数据库 / Supertag 体系（Tana/Capacities 类）
- 插件市场
- 云同步与多人协作

## 5. 能力地图（现状 → 目标）

```text
现在 (v0.9.x)
├── 本地 vault + Markdown 编辑/预览
├── Wiki 双链 / 嵌入（无系统级 backlinks）
├── 关键词搜索
├── 单篇/选区 AI（润色、摘要、标签、词条）
└── 导出 + simple-blog / 微信发布

目标 (演进后)
├── 本地 vault + Markdown 编辑/预览          [保持]
├── Wiki + 反向链接 (+ 可选邻域图)          [Phase 1]
├── 关键词 + 语义搜索 + 相关笔记            [Phase 2]
├── Ask Vault（引用跳转）+ 对话历史         [Phase 3]
├── 单篇 AI（复用同一模型栈）               [保持并整合]
├── 本地模型 / 隐私模式                     [Phase 4]
└── 导出与发布闭环                          [保持]
```

## 6. 成功指标

### 6.1 产品指标

| 指标 | 说明 |
| ---- | ---- |
| Ask Vault 周活跃 | 配置 AI 的用户中，使用库级问答的比例 |
| 引用点击率 | 答案中引用被点击打开的比例（越高说明信任与有用） |
| 语义搜索采用率 | 语义/混合搜索相对纯关键词的使用占比 |
| 反向链接使用率 | 打开面板或从 backlink 跳转的会话占比 |
| AI 写回确认率 | 生成结果被用户接受应用的比例（过高/过低都要排查） |

### 6.2 质量指标（每阶段发布前）

| 指标 | 说明 |
| ---- | ---- |
| 评测集命中率 | 20 个固定问题：检索是否召回正确笔记 |
| 引用正确率 | 答案声称的来源是否确实支持该陈述 |
| 索引重建成功率 | 损坏或手动重建后可恢复 |
| 写作无干扰 | 建索引/问答期间输入延迟不明显劣化 |
| 零未确认写回 | 无静默覆盖原文的缺陷 |

### 6.3 护栏指标

- 无 API Key 时核心写作路径完好。
- 隐私模式下无云端外泄。
- 索引目录可删；删后不影响 `.md` 本体。

## 7. 信息架构增量

在现有主界面上增量，避免变成「AI 仪表盘」：

| 区域 | 增量 |
| ---- | ---- |
| 右侧栏 | Outline 之外增加：反向链接 / 相关笔记（可切换 Tab） |
| 左侧或命令入口 | 搜索升级为「关键词 \| 语义」；Ask Vault 独立入口 |
| 设置 | AI：Provider、本地端点、Embedding、隐私模式、索引排除规则、重建索引 |
| 非目标 | 首屏堆统计、知识健康度看板、协同动态 |

## 8. 依赖与风险

| 风险 | 缓解 |
| ---- | ---- |
| 大库索引拖垮启动 | 懒建索引、后台队列、进度 UI、可暂停 |
| 幻觉与错误引用 | 强制片段约束提示词；无命中则拒答；评测集门禁 |
| 隐私顾虑 | 默认本地；发送前可见片段；隐私模式 |
| 与 Obsidian 插件生态正面撞车 | 不拼插件数量；拼「开箱即用 + 写作/发布闭环 + 确定体验」 |
| 范围膨胀成 Dify/Glean | 用 §2 原则与 §4 Phase 5 拒绝清单卡范围 |
| 中文 Wiki/文件名边界 | 链接解析与切片对中文标题、空格、大小写不敏感规则单测覆盖 |

## 9. 建议实施顺序（工程视角）

1. **链接索引服务**（Phase 1 骨架）：解析、增量、面板 UI。  
2. **文档切片 + 元数据**（为 Phase 2/3 共用）：路径、标题路径、锚点、mtime。  
3. **Embedding 提供者抽象**（本地/云）+ 向量存储。  
4. **混合检索 API**（供搜索 UI 与 Ask Vault 共用）。  
5. **Ask Vault UI + 引用跳转**。  
6. **对话历史持久化**。  
7. **本地模型与隐私模式**。  
8. **评测集与回归**（检索/引用质量）。

单篇 AI（润色等）应尽早接到同一 Provider/安全存储抽象上，避免两套配置。

| PRD 条目 | 本方案落点 |

| §7.2 更强的知识库索引与反向链接 | Phase 1 |

## 11. 决策摘要

| 问题 | 决策 |
| ---- | ---- |
| 要不要做成企业 RAG？ | 不。只做本地 vault 级 RAG。 |
| 先 Chat 还是先 Backlinks？ | 先 Backlinks + 索引骨架，再语义搜索，再 Ask Vault。 |
| AI 是否默认开启？ | 否。可选；本地模型优先叙事，云端需明示。 |
| 要不要图谱？ | 先反向链接；邻域小图可选；不做复杂全库图谱。 |
| 差异化靠什么？ | 纯 Markdown + 确定写作体验 + 发布闭环 + 开箱库级问答，而不是插件生态或协同。 |
