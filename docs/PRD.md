# M記 产品需求文档（PRD）

文档状态：1.0 发布候选  
产品版本基线：v0.9.11  
文档版本：v1.1  
更新时间：2026-07-21  
适用范围：本地优先 Markdown 桌面编辑器 `markdown-press`

## 1. 背景与机会

M記 是一款面向知识库写作的本地 Markdown 编辑器。它以本地文件夹作为知识库，不绑定云端账号或专有数据格式，强调纯 Markdown、编辑与预览一致、附件链路顺手、发布流程直接。

现有市场中，Typora 更偏单文档写作，Obsidian 更偏插件化知识管理。M記 的机会是提供一个更轻、更确定的桌面写作工具：保留 Markdown 文件的可迁移性，同时把知识库、预览、附件、导出、发布和可选 AI 辅助整合到一个低干扰的工作流里。

## 2. 产品定位

一句话定位：本地优先、面向知识库写作的 Markdown 桌面编辑器。

目标体验：比 Typora 更适合知识库连续写作，比 Obsidian 更轻量、更接近纯 Markdown 文件管理。

核心原则：

- 本地优先：笔记、图片、附件默认保存在用户选择的本地文件夹中。
- 文件透明：内容以 `.md` / `.markdown` 与常见资源文件存在，不使用封闭数据库。
- 写作优先：编辑、预览、结构导航、搜索、保存必须稳定且低打扰。
- 链路完整：从写作、附件、元信息、导出到发布形成闭环。
- 可选增强：AI、图床、发布等能力必须可配置，不影响基础写作。

## 3. 目标用户

### 3.1 核心用户

知识工作者、独立开发者、内容创作者、产品/技术写作者。他们长期使用 Markdown 管理笔记、文档、博客草稿或公众号草稿，希望数据可迁移、工具启动快、日常编辑顺手。

### 3.2 典型场景

- 使用一个本地文件夹管理长期知识库。
- 编写技术文章、产品文档、研究笔记、博客草稿。
- 在源码模式中高频编辑列表、引用、代码块、双链和附件。
- 需要在编辑器内预览公式、代码高亮、Mermaid 图表、PDF、图片或 HTML。
- 将同一篇 Markdown 导出为 PDF、长图，或发布到 simple-blog / 微信公众号草稿。
- 使用 AI 辅助生成摘要、标签、润色文本或从选区生成词条。

## 4. 问题定义

当前用户在 Markdown 写作中常见痛点：

1. 本地文件、附件、引用和发布链路割裂，写完后仍要手动整理图片、元信息和目标平台格式。
2. 编辑器与预览渲染差异大，尤其是代码高亮、数学公式、图表、附件嵌入和知识库链接。
3. 大量知识库工具功能重、设置多，普通写作路径被插件、云同步和复杂视图打断。
4. Markdown 文件虽然可迁移，但发布到博客或公众号时需要大量重复配置。
5. AI 写作能力常常与本地知识库、选区上下文、用户自定义提示词割裂。

## 5. 产品目标

### 5.1 业务目标

- 建立一个可长期使用的本地 Markdown 写作工具。
- 支持从本地知识库到导出/发布的完整工作流。
- 在不牺牲文件透明性的前提下，提供知识库管理与可选 AI 辅助。
- 形成可稳定发布 macOS / Windows 桌面版本的产品基础。

### 5.2 用户目标

- 用户可以在 1 分钟内选择一个本地文件夹并开始写作。
- 用户可以用纯 Markdown 文件完成知识库式写作，不担心迁移成本。
- 用户可以直接预览常见 Markdown 扩展内容，不需要反复切换工具。
- 用户可以将文章导出或发布到目标渠道，减少手工复制、改图链和补 metadata 的工作。

### 5.3 不做目标

- 不做云端笔记同步服务。
- 不做多人协作文档。
- 不做插件市场。
- 不做富文本优先编辑器。
- 不把 AI 作为强制依赖；没有 API Key 时基础编辑能力必须完整可用。

## 6. 成功指标

### 6.1 核心指标

- 首次打开知识库成功率：用户选择文件夹后能正常进入主界面并打开首个可编辑或可预览文件。
- 手动保存成功率：`Cmd/Ctrl + S` 后内容可靠写回原文件。
- 编辑/预览一致性问题数量：高频 Markdown 语法在编辑器、预览、导出之间无明显结构差异。
- 发布成功率：配置完整时，simple-blog / 微信公众号草稿发布能成功完成并写回必要 frontmatter。
- 崩溃/白屏率：打开知识库、切换大文件、预览复杂 Markdown 时不出现白屏或不可恢复状态。

### 6.2 体验指标

- 打开中小型知识库后，文件树、标签、编辑器、预览响应无明显卡顿。
- 列表、引用、代码块、表格等高频源码编辑行为符合主流 Markdown 写作预期。
- 设置项改变后可立即理解其影响，不需要重启或查文档。

## 7. 产品范围

### 7.1 当前版本必须覆盖

- 本地知识库打开、恢复、文件树浏览。

- 预览渲染：标题、列表、表格、代码高亮、数学公式、Mermaid、任务列表、脚注、图片、附件嵌入。
- Outline 结构导航。
- 当前文件搜索与侧边栏搜索。
- Wiki 双链和嵌入链接。
- 图片粘贴、本地资源目录、附件清理。
- PDF 导出、纯文本导出、长图分享。
- simple-blog 发布。
- 微信公众号草稿发布。
- 可选 AI 辅助：润色、摘要、标签、选区生成词条。
- 主题、字体、字号、语言、快捷键、metadata、发布、图床、更新等设置。
- macOS / Windows 桌面安装包与更新检查。

### 7.2 后续版本可扩展

- 更完整的图床批量迁移。
- 更强的知识库索引与反向链接。
- 更细粒度的 Markdown 风格预设。
- 更完整的发布渠道。
- 大型知识库性能优化。
- AI 与本地知识库检索结合。

## 8. 核心用户流程

### 8.1 首次使用

1. 用户启动应用。
2. 应用展示选择知识库入口。
3. 用户选择一个本地文件夹。
4. 应用读取文件树，必要时初始化示例笔记。
5. 应用打开最近文件或第一个可打开文件。
6. 用户进入主编辑界面。

验收标准：

- 未选择文件夹时不进入空白主界面。
- 读取失败时给出明确错误提示。
- 首次打开不会破坏用户原有文件。

### 8.2 日常写作

1. 用户从侧边栏打开 Markdown 文件。
2. 文件以标签页形式打开。
3. 用户在编辑、预览、分屏模式之间切换。
4. 用户使用快捷键编辑列表、引用、代码块、双链、图片。
5. 应用自动保存，用户也可以手动保存。
6. 用户通过 Outline 或搜索定位内容。

验收标准：

- 切换标签时保留每个文件的内容状态。
- 保存失败时必须提示，不能静默丢失。
- 附件类资源只能预览，不误进入 Markdown 编辑状态。

### 8.3 附件与图片

1. 用户粘贴或引用图片。
2. 应用按设置保存到资源目录，或上传到图床并插入公网 URL。
3. 预览侧展示图片。
4. 用户可以清理未引用附件。

验收标准：

- 本地图片路径与知识库相对引用保持稳定。
- 图床上传失败时提示清楚，并按设置回退或停止。
- 清理附件前必须避免删除仍被引用的资源。

### 8.4 导出与分享

1. 用户打开一篇 Markdown。
2. 用户选择导出 PDF 或生成长图。
3. 应用使用当前预览样式、字体、代码高亮和附件解析生成结果。
4. 用户保存文件或复制长图。

验收标准：

- 导出结果与预览语义一致。
- 包含 Mermaid、代码块、公式、图片的文章可正常导出。
- 极长文档不应导致应用白屏或不可恢复。

### 8.5 发布到 simple-blog

1. 用户配置博客仓库、公开站点地址、GitHub Token。
2. 用户点击发布入口并选择 simple-blog。
3. 应用保存当前笔记，处理 frontmatter 与图片资源。
4. 应用同步 `posts/` 和图片到目标仓库。
5. 应用写回 `is_publish`、`link` 等字段。

验收标准：

- 配置缺失或格式错误时阻止发布并提示。
- 发布失败时不写入伪造的成功状态。
- 修改 `slug` 不应意外改变仓库内文件名，除非产品明确调整。

### 8.6 发布到微信公众号草稿

1. 用户配置公众号 AppID / AppSecret。
2. 用户选择微信公众号草稿发布。
3. 用户填写或确认标题、作者、摘要、原文链接，并选择封面。
4. 应用上传正文图片和封面图。
5. 首次发布创建草稿，再次发布更新已有草稿。
6. 应用写回 `wechat_draft_media_id`。

验收标准：

- 未配置凭据或未选封面时不能提交。
- 正文中的本地图片必须替换为微信可访问图片。
- 已有 `wechat_draft_media_id` 时默认走更新草稿。

### 8.7 AI 辅助

1. 用户在设置中选择 AI Provider 并配置 API Key、模型与提示词。
2. 用户对当前文档或选区触发 AI 功能。
3. 应用展示摘要、标签、润色结果或生成词条内容。
4. 用户决定是否使用结果。

验收标准：

- 未配置 Key 时不影响基础编辑。
- AI 请求失败时提示清楚，不破坏当前内容。
- 生成内容需要用户确认，不应静默覆盖原文。

## 10. 非功能需求

### 10.1 性能

- 中小型知识库打开后，文件树与编辑器应在可感知范围内完成初始化。
- 大文件编辑不能因为预览或语法高亮导致内容丢失。
- 预览渲染、Mermaid、代码高亮、长图导出需要有失败兜底。
- 搜索与 Outline 更新不能明显阻塞输入。

### 10.2 可靠性

- 保存、发布、导出等写操作必须有成功/失败反馈。
- 敏感配置不得进入普通持久化状态。
- 应用重启后应恢复用户最后选择的知识库和必要 UI 状态。
- 版本升级不能破坏旧设置；已移除设置需要清理或兼容。

### 10.3 安全与隐私

- 默认所有笔记和附件保存在本地。
- GitHub Token、公众号 AppSecret、AI API Key、图床密钥必须通过安全存储保存。
- 发布、图床、AI 请求必须由用户配置后显式触发。
- 不应在未经用户确认的情况下上传本地文件内容。

### 10.4 兼容性

- 支持 macOS 和 Windows 桌面环境。
- 浏览器开发环境可用于基础调试，但桌面专属功能需要明确降级或提示。
- Markdown 文件应保持通用语法，不写入工具私有格式作为必要依赖。

### 10.5 可维护性

- 编辑器行为、预览渲染、导出发布、设置持久化应保持模块边界清晰。
- 对已修复的问题应补充对应测试用例，避免回归。
- 新增对外可见功能需要先确认产品范围，再进入实现。

## 11. 信息架构

主界面：

- 左侧：知识库文件树、搜索入口、附件/文件管理。
- 顶部：标签页、工具栏、视图切换、发布/导出/AI 入口。
- 中央：编辑器、预览或分屏区域。
- 右侧：Outline 结构导航。
- 弹层：设置、发布目标选择、公众号草稿、长图分享、确认类操作。

设置页：

- Interface：语言、主题、UI 字体等。
- Editor：编辑器字体、字号、换行、格式化、列表行为等。
- Metadata：frontmatter 默认字段。
- Publishing：simple-blog 与微信公众号配置。
- Image Hosting：图床与粘贴行为配置。
- AI：Provider、模型、Key、提示词。
- Shortcuts：快捷键说明。
- Updates / About：版本与更新检查。

## 12. 数据与状态

### 12.1 本地文件

- Markdown 内容直接写入用户知识库目录。
- 图片和附件默认进入资源目录。
- 回收类操作进入配置的回收目录。

### 12.2 应用状态

- 普通设置可持久化到本地状态。
- 敏感字段只进入安全存储。
- 打开的标签、当前文件、最近知识库等作为本地体验状态恢复。

### 12.3 frontmatter 字段

常用字段：

- `title`：文章标题。
- `aliases`：别名。
- `slug`：公开链接 slug。
- `link`：发布后公开链接。
- `status`：写作状态。
- `is_publish`：simple-blog 发布标识。
- `wechat_draft_media_id`：微信公众号草稿标识。

## 13. 验收标准总览

### 13.1 P0 验收

- 用户可以打开本地知识库并编辑 Markdown。
- 用户可以保存文件，重启后内容仍在原路径。
- 用户可以在编辑、预览、分屏模式之间切换。
- 用户可以预览核心 Markdown 内容和本地图片。
- 用户可以导出 PDF。
- 用户可以配置并发布到 simple-blog。
- 敏感配置不进入普通 localStorage。
- 应用不会因打开附件类资源而进入错误编辑状态。

### 13.2 P1 验收

- Outline、搜索、Wiki 链接、附件清理可正常工作。
- Mermaid、公式、代码高亮在预览与导出中可用。
- 长图生成可预览、保存或复制。
- 微信公众号草稿发布支持首次创建和再次更新。
- AI 功能在配置完整时可用，失败时不影响编辑。

## 14. 发布策略

### 14.1 版本节奏

- Patch 版本：修复保存、编辑器行为、预览、导出、发布等明确 bug。
- Minor 版本：新增发布渠道、图床能力、AI 能力、样式预设等功能。
- Major 版本：涉及文件结构、核心编辑器架构或兼容性边界变化。

## 15. 风险与应对

| 风险                       | 影响                 | 应对                                       |
| -------------------------- | -------------------- | ------------------------------------------ |
| 编辑器行为与用户习惯不一致 | 高频写作受阻         | 对列表、缩进、回车、选区操作建立回归测试   |
| 预览与导出不一致           | 用户对输出不信任     | 共用渲染准备链路，复杂内容补样例验证       |
| 发布链路依赖第三方接口     | 失败率与错误排查复杂 | 配置校验、超时、错误本地化、失败不写回状态 |
| 敏感配置泄露               | 严重安全风险         | 敏感字段统一安全存储，普通持久化中清理     |
| 大知识库或大文件性能问题   | 卡顿、白屏、内容丢失 | 分层缓存、延迟渲染、关键路径测试           |
| 功能范围扩大导致产品变重   | 偏离轻量定位         | 新功能默认可选，避免强制进入主写作路径     |
